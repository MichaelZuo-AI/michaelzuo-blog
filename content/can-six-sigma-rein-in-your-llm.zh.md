过去两个月我几乎每天都在和 Claude Code 打交道。有一件事一直困扰着我。

同样的输入,同样的 prompt,两次 run 出来的代码可以差得离谱。有时候一把过,干净、准确、可以直接合进去。有时候 feature 做错了方向、做了一半、或者代码里从一开始就藏着 bug。你没办法预判下一份输出会是哪一种。

能想到的办法我都试了。把 prompt 写得更严,把 context 裁得更干净,多塞一些示例。平均水平确实上去了。那种"最好一次"和"最差一次"之间的差距,一点没变。

琢磨了一阵,我大概看清了怎么回事。

Prompt 工程调的是平均水准,不是输出的一致性。它能把中位数往右挪,挪不了分布的宽度。要让输出稳下来——我说的是"稳"而不是"好"——得换一套工具。

这套工具,工厂里已经用了一百年。

## 我们都在把系统性问题当运气不好处理

1924 年,美国工程师 Walter Shewhart 对"过程中出现的偏差"做了一个区分,后来成为整个统计质控领域的起点。

他说,偏差分两种。一种叫*特殊原因*——某一次出问题,是因为某件具体的事:一批次原料不合格、一个操作员疏忽、一台设备坏了。这种问题能指认、能修、修完就消失了。另一种叫*共同原因*——它就是系统本身的一部分。不属于哪一次,不归咎于谁,它是你设计出来的这套流程每次都会产生的噪声。

再回头看 Claude Code 那件事。输出不理想,你的第一反应是什么?重新跑一次。调一调 prompt 再试。心里想"这次它状态不太好"。

这个反应,本质就是把共同原因看成了特殊原因。那种"AI 这次没理解我"的感觉,其实并不是偶发的——它每次都在那里。只是多数时候结果落在你能接受的那一段,你没在意;等它偶尔跌到很差的一端,你才开始抱怨这一次。

你一直盯着的是分布两端的尾巴,却以为自己在看意外事件。

所以要想真把这件事处理好,首先得把问法换了。不是"这次运气怎么样",是"这套过程本身的能力到底有多稳"。这,是 Six Sigma 要教我们说的话。

## Six Sigma 不是来帮你消灭 bug 的

这个方法论的名字其实很老实,它把核心直接写在招牌上了。

"六个标准差"——也就是说,你的过程产出的 ±6σ 范围内的值,都应该落在你定义的"合格区间"里。那个流传很广的"一百万次里只允许 3.4 次不合格",不是谁凭感觉定的目标,是从这个几何关系直接算出来的。整套工具——控制图、过程能力指数、实验设计——统统围绕两件事在转:测 σ,缩 σ。

视角一旦转到这里,很多东西就不一样了。**所谓"缺陷",不是指 bug,而是指变异本身。** "合格区间"由你定义——代码得实现需求、风格得统一、架构得能融进现有代码。你要做的,是让每次的输出都落在这个区间里。

这么看,harness 工程其实既不是在修 prompt,也不是在等更好的模型。它处理的是一个*过程能力*的问题。

而处理过程能力问题的第一步,从来都是同一个动作:把变异拆成几类,分别对付。

| 变异的来源 | 常见症状 |
|---|---|
| 模型本身的随机性 | 同 prompt 不同 run,结果波动 |
| 输入层的变异 | Prompt 措辞飘、上下文被污染 |
| 过程层的变异 | Agent 走不同分支、工具调用出错 |
| 格式/风格层的变异 | 架构不一致、代码格式漂移 |
| 需求落地层的变异 | 代码没做到需求 |

## 五类变异,一类一类看

### 模型本身的随机性

这类最好识别。

模型采样 token 本身就是概率过程,就算你的输入一字不差,输出也会不一样。能动的旋钮有两个:一个是温度,直接影响分布的宽度——温度压低,输出就收紧;另一个是 K-of-N 采样,让它生成 N 份候选,挑其中最好的那一份。

AI 圈子最愿意承认的就是这一类变异。但我这两个月观察下来,它在真实痛点里的占比,其实被高估得很严重。很多被归咎给"这次采样不给力"的情况,拆开看根本不是采样问题,而是底下四类之中有一类穿着采样的外衣在捣乱。

所以这个杠杆该用就用,但别指望它一个人就能扛下来。K-of-N 还有个要当心的地方:"从 N 份里挑最好的",前提是你得有个靠谱的"挑法"。对代码来说,这个前提成立得很脆弱,待会儿会说到为什么。

### 输入层的变异

同一个需求,换个说法递给模型,吐出来的代码就不一样了。上下文里混进了杂音——可能是前几轮对话里没清理掉的遗留,也可能是 RAG 检索塞进来的边角料。同一个任务,实际被模型看到的已经不是同一个任务了,输出分布自然跟着走样。

这一层的应对方法,圈子里已经摸索了好几年。用模板把 prompt 的措辞锁住,把 context 里不相关的部分裁掉,把"最小必要上下文"当作原则来执行。Prompt 工程这两三年做的大部分事,其实都在这里。

Six Sigma 在这一格没加什么新工具,只是给它换了个更严谨的名字——*输入稳定性*。没有新东西。但你一旦把它当成五类之一、而不是"问题的全部",你花时间的地方就会很不一样。

### 过程层的变异

Agent 的循环会分叉。每一步,模型都在做决定:调哪个工具、搜索什么、跳过什么。同一个任务跑两次,完全可能走出两条完全不同的路径——读到的文件不一样,形成的问题理解不一样,甚至在下笔之前对任务的定性就不一样。

这一层能做手脚的地方不少:强制把 loop 收敛到某个固定的骨架;给工具列白名单,名单外的一律禁用;在关键节点放验证 gate,通不过就绕回去重走。

"收敛"是最直白的杠杆——也是最危险的一个。Agent 会产生变异,根源就在于它会在每一步做选择。而这种选择能力,恰恰是它在你没设计的情况下还能解决问题的来源。你把收敛压得越狠,压下去的东西,可能不只是变异。这个取舍稍后会回头讨论。

### 格式/风格层的变异

不同次跑出来的架构选型、代码格式、命名习惯、文件怎么放,可能都不一样。底层逻辑也许是对的,但外面这层看起来像不同的人在写。

这一类最好处理,因为它的应对基本已经是纯工程问题。Linter、validator、各类 schema 约束、发现不合规就让模型在更紧的框架里重写——这套组合拳,工具链社区已经打得很顺手了。

Six Sigma 在这里也没拿出新牌,它只是给这件事一个名字:这也是变异,而且它已经被压得差不多了。好,翻篇,下一类。

### 需求落地层的变异

最痛的就是这一类。

代码能编译。测试也能过(你没写的那些)。看起来就是一个做完的 feature。直到你把它放回需求文档旁边,一条一条对,发现它做的,是相似但不是同一件事。可能方向偏了一丢。可能只做了一半。可能注释里*一本正经*地写着需求,底下实际做的是另一回事。这种"不可用",linter 根本抓不到。

这才是前面那个"把共同原因当特殊原因"最戏剧化的表现。你的反应照旧——重写 prompt,心里念叨"这次它怎么就是不懂我"。但多数情况下,这不是这次的运气问题,是你这套 harness 设计本身的漏洞:它从头到尾,就没有用一种"可以被验证的形式"把需求强制送进流程。需求躺在你脑子里、压在一行 prompt 里、或者躺在一份模型假装读过的文档里。从"需求 → 测试 → 代码 → 验证"的回路,没有任何一个环节真正落地过。

这一类的对策都不轻,运维代价也不小:把验收标准(AC)前置到 prompt 里;让模型先写一份计划,你批准之后再动代码;测试先写,把测试本身当作契约(也就是 TDD 那套);跑验证 gate——要么跑单元测试,要么让另一个 LLM 对着规格打分;两个 agent 互相审,一个出方案,一个查漏。

每一条,本质都是在流程的某个固定节点上,把需求以"可以被机器验证"的形态摆出来。但这些手段到底能不能把分布真正收窄,是另一个问题——也正是我下面要挑出来讲的三条假设之一。

| 变异的来源 | 常见症状 | Harness 层可以动的杠杆 |
|---|---|---|
| **模型本身的随机性** | 同 prompt 不同 run,结果波动 | 温度、采样、K-of-N |
| 输入层的变异 | Prompt 措辞飘、上下文被污染 | Prompt 模板化、context 裁剪 |
| **过程层的变异** | Agent 走不同分支、工具调用出错 | 流程收敛、工具白名单、验证 gate |
| 格式/风格层的变异 | 架构不一致、代码格式漂移 | Linter、validator、schema、再生 |
| **需求落地层的变异** | 代码没做到需求、bug 扎堆出现 | 规格前置、plan-before-code、TDD、验证 gate、双 agent 交叉校验 |

加粗的这三行,是我自己最没把握的三条。为什么没把握,下面一条一条说。

## 我最没把握的三条假设

**1. 用一个 LLM,去给另一个 LLM 的输出打分,能不能真的把需求落地变异压下去?**

打分的这个 LLM,自己就是个高变异的家伙。你用一个变异大的工具,去测量另一个变异大的工具的输出——结果很可能只是把变异从 A 转到了 B。

Six Sigma 对这件事有一个前置的把关步骤,叫*测量系统分析*(MSA)。核心要求是:你用来测量的那个工具,精度得比你想测的那个过程细一个数量级,否则你画的控制图,读出来的其实不是过程的变异,是你测量工具自己的噪声。LLM 当裁判,很可能还没跨过这道门槛。

**2. K-of-N 采样能不能真的降低代码输出的变异?**

K-of-N 在有"标准答案"的任务上确实管用——数学题、分类题。但代码不一样,同一个需求,几种不同的实现都可能是"对"的。要从里面挑一份最好,你还得有个仲裁者。绕了一圈,又回到了第一条。而且 K-of-N 在代码任务上的实际效果,远没有它在那些成名 benchmark 上那么亮眼。

**3. 流程收敛能不能在"不削弱能力"的前提下,把过程变异压下去?**

Agent 循环之所以会产生变异,根子就在于模型在每一步都在选分支。而分支这件事,本身也是它在面对你没预设好的情况时仍然能想办法解决问题的能力来源。你把收敛勒得越紧,压下去的东西里,不会只有变异——很可能一起被压下去的,还有能力。

Six Sigma 对这种张力也有名字,叫*过程稳定性 vs 过程能力*。如果你只围绕稳定性来优化,相当于你把 harness 训练成了一个更守规矩、但也更笨的工程师。

---

所以,这篇文章是一份设计,不是一份结论。接下来几周,我打算在自己的 harness 上,针对每一类变异跑一到两个最小规模的干预——调温度、把 prompt 模板化、把 agent 流程收敛一点、把已经在跑的 linter 继续放着,再加上需求落地那几条比较贵的动作——然后看输出的分布到底会不会真的变窄。

上面那三条假设,是我会盯得最紧的几条。如果它们崩了,这个框架会崩得挺有意思,到时候我带着碎片回来写续集。

**Prompt 工程调的是平均水平。要让输出稳下来,答案不在 prompt 里,在工厂的流水线上。**
