过去两个月我一直在和 Claude Code 密集协作。同样的输入,同样的 prompt,不同次运行——输出质量从很差到很高都有。有些 run 产出干净、聚焦、把事情做完的代码;另一些 run 产出的代码没做到需求、实现错了方向,或者把 bug 一并烘焙进来了。分布足够宽,宽到我无法预期我会拿到什么。

我一直在打磨 prompt。更清晰的指令、更精简的 context、更明确的例子。输出均值确实在变好。分布宽度没有。

我这才意识到自己一直在用错误的工具处理错误的问题。Prompt engineering 是一种均值工具。它把分布的中心往右上挪。但变异——那个*宽度*——是另一种东西。它归另一套方法管。

而那套方法,一百年前就被制造业的工程师们琢磨透了。

## 你一直在把共因当特殊因处理

1924 年,Walter Shewhart 把过程变异切成了两类。*特殊因*——可归咎到某个具体事件,可以指出来、修掉、翻篇。*共因*——烘焙在过程设计里,每次都在发生,永远无法归咎到任何单独事件。

当 Claude Code 递给你一份平庸的输出时,你怎么做?重新 prompt。再试一次。把它归为这次运气差。那就是把共因当特殊因处理的反射。

多数那种"这次 AI 没理解我"的时刻,其实是每次都在发生的事。你只是在结果落到分布差的那一端时才注意到。

要收窄分布,我们需要的是*"这套过程本身能力如何"*的语言,而不是*"这次运气怎么这么差"*的语言。那是 Six Sigma 的语言。

## Six Sigma 的真正内核不是消灭 bug

名字本身就给了答案。*六西格玛*的意思是 ±6 个标准差必须装进你的规格边界之内。那个著名的"百万机会缺陷数 3.4"的目标不是凭空挑的——它是从那个几何关系直接推出来的。工具箱里的每样东西——控制图、过程能力指数、实验设计——都存在于两件事:测量 σ,缩小 σ。

一旦看清这一点,framing 就变了。**缺陷就是变异本身。** 规格边界是你怎么定义*可接受*——代码实现了需求、风格一致、架构跟代码库合得上。工作就是把分布关进那些边界里。

Harness engineering 不是 prompt engineering 问题,甚至也不是模型质量问题。它是*过程能力*问题。

第一步,是把变异拆成来源。

| 变异来源 | 典型症状 |
|---|---|
| 模型随机性 | 同 prompt 不同 run 结果波动 |
| 输入变异 | Prompt 漂移、上下文被污染 |
| 过程变异 | Agent loop 分支、工具调用失败 |
| 格式/风格变异 | 架构不一致、格式漂移 |
| 需求落地变异 | 代码没按需求实现 |

## 把变异分解成来源

### 模型随机性

最容易辨认的一类。模型按概率采样 token,即便输入固定,输出也会不同。Temperature 直接控制分布宽度——温度越低分布越窄。K-of-N 采样——生成 N 个候选,挑最好的那个——是另一个常见动作。

这是 AI 社区最愿意承认的变异来源。它也是"真实痛点"占比最容易被*高估*的一类。很多我们归咎到"这次采样差"的情况,其实不是采样噪声。而是另外四类中的某一类,只是装扮成了采样噪声。

用它,但别指望它能独自解决多少问题。而且 K-of-N 要看仔细:从 N 个候选里挑最好的,预设了你有一个好的挑法。对代码来说,这个预设以有趣的方式失效。稍后再说。

### 输入变异

同一个意图,不同 prompt 表述,产出不同输出。Context 被早期对话的无关信息、或者过度积极的检索结果污染了。同一个任务变成了不同的任务,输出分布也跟着漂。

Harness 层的杠杆已经很成熟:prompt 模板化钉住表述,context 裁剪移除噪声,最小必要上下文原则保持输入信号密度。这是过去三年 prompt engineering 社区一直在做的事。

Six Sigma 在这里加的主要是一个更严谨的名字:*输入稳定性*。这一行没什么新东西。但把它看作五类变异之一、而不是"整个问题的全部",会更清楚一点。

### 过程变异

Agent loop 会分支。模型每一步都在决定调哪个工具、搜什么、跳过什么。同一个任务的两次 run 可能走完全不同的执行路径、访问完全不同的文件、在写下第一行代码之前已经建立起完全不同的问题心智模型。

Harness 层杠杆包括强制收敛流程(loop 必须经过特定形状)、工具白名单(名单外的不能调)、验证 gate(通过或重走)。

显而易见的杠杆是流程收敛——它也是最危险的。Agent 会产生变异,恰恰是因为它在分支。而分支本身是它解决你没预料到的问题的能力来源。收敛是用其中一个去换另一个。稍后回来说。

### 格式/风格变异

不同 run 产出不同的架构选择、不同的格式、不同的命名约定、不同的文件布局。底层逻辑可能一样,但表层在漂。

这是表格里最*机械*的一类。Harness 已经有好用的杠杆:linter、validator、schema 约束、发现违规就让模型在更紧的框架里再试一次的再生策略。

Six Sigma 在这里没加新工具——工程工具社区已经把格式约束当作一个基本解决的问题处理一段时间了。它只是加了一个命名:这也是变异,而且已经在被压缩。很好。翻篇。

### 需求落地变异

最痛的那一类。代码能编译。输出看起来像一份完成的 feature。然后你把它放在需求旁边对比,发现它实现的是稍有不同的东西。或者只实现了一部分。或者它在注释里*描述*了需求,然后做了别的事。Feature 不可用——但不是任何 linter 能抓出来的那种不可用。

这是 Section 2 的最大音量——整个框架里*把共因错当特殊因*最戏剧性的一行。反射是重新 prompt:*"这次模型没理解我。"* 但大部分情况,是 harness 设计的必然输出——它从来没有把需求规格以可验证的形式强制进流程。规格活在你脑子里、或一句 prompt 里、或一份模型假装读了的文档里。路径上没有任何一处强制执行了"从规格到测试到代码到验证"的闭环。

这一类的杠杆最重、运营成本最高:规格前置 prompt(AC 骑在 prompt 里)、plan-before-code(模型先写计划,你批准,再写代码)、TDD 式*"测试即规格"*(测试先于实现,测试即契约)、验证 gate(跑测试、或让 LLM-as-judge 对照规格打分)、双 agent 交叉校验。

每一条都是在让规格以可验证的形式、在流程的某个特定节点上显形。这些能不能真的把分布收窄,是另一个问题——也是我马上要标出的三条假设之一。

| 变异来源 | 典型症状 | Harness 层杠杆 |
|---|---|---|
| **模型随机性** | 同 prompt 不同 run 结果波动 | Temperature、采样、K-of-N |
| 输入变异 | Prompt 漂移、上下文被污染 | Prompt 模板化、context 裁剪 |
| **过程变异** | Agent loop 分支、工具调用失败 | 流程收敛、工具白名单、验证 gate |
| 格式/风格变异 | 架构不一致、格式漂移 | Linter、validator、schema、再生 |
| **需求落地变异** | 代码没按需求实现、bug 扎堆 | 规格前置、plan-before-code、TDD、验证 gate、双 agent 交叉校验 |

加粗的三行,是我最没把握的三条。原因如下。

## 我最怀疑的三条假设

**1. "LLM-based 验证 gate 能收窄需求落地变异。"**

验证器本身就是一个高变异的 LLM。用一个变异高的工具去压另一个变异高的工具的输出,很可能只是把变异从 A 搬到 B。Six Sigma 恰好有一个前置步骤——*测量系统分析*。测量工具的分辨率必须比被测过程的变异小一个量级,否则控制图读出来的就是测量噪声。LLM-as-judge 大概还没过这道门槛。

**2. "K-of-N 采样能降低代码输出变异。"**

K-of-N 在有"标准答案"的任务上有效——数学、分类。代码里,多条输出都可能"对"。挑最好的需要一个仲裁者,又踢回到假设 1 的坑里。而且 K-of-N 在代码任务上的经验证据,远比在它最初成名的那些 benchmark 上要弱。

**3. "流程收敛能在不损失能力的前提下压过程变异。"**

Agent loop 产生变异,恰恰是因为模型在选分支,而分支本身是它解决你没规划到的问题的能力来源。过度收敛,可能把变异和能力一起压下去。Six Sigma 在这件事上也有一个名字——*过程稳定性 vs 过程能力*。只优化稳定性,等于把 harness 训练成一个更笨的工程师。

这是一份设计,不是结论。接下来几周,我会在自己的 harness 里,针对每一条变异来源各跑一到两个最小干预——温度控制、prompt 模板化、流程收敛、已经在跑的 linter,以及需求落地变异那几条昂贵的——看输出分布会发生什么。上面那三条假设是我最紧盯的。如果它们垮了,这个框架会以有意思的方式垮掉,届时我带着碎片回来。

**Prompt engineering 调的是均值。收窄分布,得去生产线上找答案。**
