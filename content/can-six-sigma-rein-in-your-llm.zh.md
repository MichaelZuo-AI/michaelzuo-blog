过去两个月我每天都在用 Claude Code。有一件事越用越在意:同样的输入,同样的 prompt,跑两次,出来的代码完全不是一个水平。有时候很好。有时候没做到需求,或者只做了一半,或者一上来就带着 bug。我压根没法预测下一次会是哪一种。

试过几种能想到的办法。更严的 prompt,更干净的 context,再多塞几个例子。均值上去了。分布还是那么宽。

过了一阵子才看清:prompt engineering 调的是均值,不是分布。要把范围收窄,得换一套工具。

而那套工具,在工厂里已经用了一百年。

## 你一直在把共因当特殊因处理

1924 年,Walter Shewhart 把过程里的变异分成了两类。一类叫*特殊因*——来自某个具体事件,能指出来,能修掉。另一类叫*共因*——是过程设计本身的一部分,每次都在发生,但谁也不能被单独归罪。

Claude Code 给你一份不理想的输出,你会怎么做?重跑一次。改 prompt。告诉自己这次运气差。那就是在把共因当特殊因处理。

那种"这次 AI 没懂我"的感觉,多半不是偶发的。它每次都在。只是大部分时候结果落在分布中段,你没感觉;等它偶尔落到差的那一端,你才注意到。

要收窄分布,语言得先换。不是"这次怎么这么背",而是"这个过程的能力到底怎样"。这是 Six Sigma 的语言。

## Six Sigma 的内核不是消灭 bug

名字本身就说明了一切。*六西格玛*的意思是 ±6 个标准差要装进规格边界之内。那个著名的"百万机会缺陷 3.4"不是凭空定的目标——它是从那个几何关系直接推出来的。整套工具箱——控制图、过程能力指数、实验设计——都只在做两件事:测 σ,缩 σ。

从这个角度看,**缺陷就是变异本身。** 规格边界由你定义——代码实现了需求、风格连贯、架构跟已有代码合得上。工作就是让输出分布待在这些边界里。

所以 harness engineering 其实不是 prompt 的问题,也不是模型质量的问题。它是个*过程能力*问题。

要做的第一件事,是把变异拆开看。

| 变异来源 | 典型症状 |
|---|---|
| 模型随机性 | 同 prompt 不同 run 结果波动 |
| 输入变异 | Prompt 漂移、上下文被污染 |
| 过程变异 | Agent loop 分支、工具调用失败 |
| 格式/风格变异 | 架构不一致、格式漂移 |
| 需求落地变异 | 代码没按需求实现 |

## 把变异拆成来源

### 模型随机性

最容易看见的一类。模型采样 token 时是概率性的,哪怕输入完全一样,输出也会变。Temperature 直接影响分布宽度,温度低分布就窄。K-of-N(生成 N 个候选挑一个最好的)是另一种常见做法。

这是 AI 社区最愿意承认的变异来源。它也是真实痛点里被高估得最多的一类。很多我们骂"这次采样差"的情况根本不是采样问题,是另外四类里的某一类伪装成它。

可以用,但别指望它单独能解决多少事。K-of-N 还有一个要小心的地方:从 N 个里挑最好的,前提是你有一个好的挑法。对代码来说,这个前提破得很有意思,后面会说到。

### 输入变异

同一个意图换个 prompt 表述,输出就不一样。Context 被早前对话里的无关信息、或者太积极的检索结果污染了。同一个任务于是变成了不同的任务,输出分布跟着挪。

Harness 层的杠杆已经挺成熟:prompt 模板把表述钉住,context 裁剪把噪声去掉,最小必要上下文让输入信号更密一点。过去三年 prompt engineering 社区做的大部分事都在这里。

Six Sigma 在这一行主要是给它起了个更严的名字:*输入稳定性*。没什么新内容。但把它当成五类之一、而不是"问题的全部"来看,你花时间的地方会不一样。

### 过程变异

Agent loop 会分支。模型每一步都在决定调哪个工具、搜什么、跳过什么。同一个任务的两次 run,可能走完全不同的路径,读完全不同的文件,在写下第一行代码之前就已经对问题有了完全不同的判断。

杠杆包括:强制流程收敛(loop 必须经过某个固定结构)、工具白名单(名单外不能调)、验证 gate(过或者不过)。

最直接的杠杆是收敛——也是最危险的一个。Agent 产生变异,*就是*因为它在分支。而分支,同时也是它解决你没想到的问题的能力所在。收敛一项,就是拿这个换那个。等一下会专门讲。

### 格式/风格变异

不同次 run 产出的架构选择、格式、命名、文件布局都可能不一样。底层逻辑可以是对的,但表层在漂。

这一类是表格里最机械的一类。Harness 已经有成熟的杠杆:linter、validator、schema 约束,还有发现违规就让模型在更紧的约束里重来一次的再生策略。

Six Sigma 在这里没带来新东西——工程工具社区早就把格式约束当作一个基本解决的问题处理了。它只是加了一个名字:这也是变异,而且已经被压住了。好的。下一类。

### 需求落地变异

最痛的那一类。代码能编。输出看起来像一个做完的 feature。你把它拿到需求旁边一对,发现实现的是稍有不同的东西。或者只实现了一部分。或者注释里*写*了需求,但实际做的是另一件事。Feature 不可用,而这种不可用 linter 抓不出来。

这就是整个框架里把共因错当成特殊因的最极端的一行。反射动作还是重跑 prompt:*"这次模型没懂我。"* 但多数情况下,这是 harness 设计的必然产物——harness 从来没有用一种可验证的形式把规格强制进流程。规格在你脑子里,在一句话的 prompt 里,或者在一份模型假装读过的文档里。从规格到测试到代码到验证的闭环,路径上没有一处在强制执行。

这一类的杠杆最重,代价也最大:规格前置 prompt(把 AC 写进 prompt,而不是留在脑子里);plan-before-code(模型先写计划,你批准,再写代码);TDD 式的*"测试即规格"*(测试先写,测试就是契约);验证 gate(跑测试,或者让一个 LLM 以规格为参照去 judge);双 agent 交叉校验。

每一条都是在逼迫规格以可验证的形式、在流程的某个具体节点上显形。这些手段能不能真的收窄分布,是另一个问题——也是我马上要标出来的三条假设之一。

| 变异来源 | 典型症状 | Harness 层杠杆 |
|---|---|---|
| **模型随机性** | 同 prompt 不同 run 结果波动 | Temperature、采样、K-of-N |
| 输入变异 | Prompt 漂移、上下文被污染 | Prompt 模板化、context 裁剪 |
| **过程变异** | Agent loop 分支、工具调用失败 | 流程收敛、工具白名单、验证 gate |
| 格式/风格变异 | 架构不一致、格式漂移 | Linter、validator、schema、再生 |
| **需求落地变异** | 代码没按需求实现、bug 扎堆 | 规格前置、plan-before-code、TDD、验证 gate、双 agent 交叉校验 |

加粗的那三行,我最没把握。原因如下。

## 我最没把握的三条假设

**1. "LLM-as-judge 的验证 gate 能收窄需求落地变异。"**

验证器本身就是个高变异的 LLM。用一个变异高的工具去压另一个,结果很可能只是把变异从 A 搬到 B。Six Sigma 对这件事有个前置步骤——*测量系统分析*。测量工具的分辨率得比被测过程的变异细一个量级,否则控制图读出来的就是测量噪声。LLM-as-judge 大概还没过这道门槛。

**2. "K-of-N 采样能降低代码输出的变异。"**

K-of-N 在有"标准答案"的任务上好用——数学题、分类题。代码不一样,几个不同的实现都可能是"对"的。挑最好的又得有个仲裁者,兜一圈又回到第一条。而且 K-of-N 在代码任务上的经验数据,比它一炮而红的那些 benchmark 弱得多。

**3. "流程收敛能在不损失能力的前提下压过程变异。"**

Agent loop 产生变异,恰恰是因为模型在选分支。而分支,是它解决你没设计进去的问题的能力所在。过度收敛,压下去的很可能不只是变异,还有能力。Six Sigma 对这个张力也有个名字——*过程稳定性 vs 过程能力*。只优化稳定性,就是把 harness 训练成一个更笨的工程师。

这是一份设计,不是结论。接下来几周,我会在自己的 harness 里,针对每一类变异都跑一到两个最小干预——温度、prompt 模板、流程收敛、本来就在跑的 linter、以及需求落地变异那几条贵的——然后看输出分布会发生什么。上面那三条假设,是我最紧盯的。如果它们倒了,这个框架会倒得挺有意思,到时候我带着碎片回来。

**prompt engineering 调的是均值。要收窄分布,得去生产线上找答案。**
