人为什么总喜欢讲因果? 不是因为人类天生爱讲故事,而是因为行动需要因果。预测只能告诉你下一步大概会发生什么。因果告诉你:如果我改了这里,结果会不会跟着变。工程工作靠的是后者。

所以当你把 Claude Code 拉进写代码这件事时,脑子里自然会浮出一张很短的图:SPEC 写清楚,模型照着做,正确代码就出来。现实却很少这么配合。需求会漏,边界条件会掉,代码看起来像样,跑起来还是不对。

这不是因为因果思维失效了。是因为你画的那张因果图太短了。

## 人为什么总想先找原因

人碰到问题时,本能不是先做概率分布,而是先问:"到底是哪一环出了问题?" 这个习惯很合理。因为你真正要做的不是解释世界,而是干预世界。你要知道下一把该拧哪颗螺丝。

软件里这种倾向更强。我们天天都在问反事实问题:如果我改 prompt,结果会不会变; 如果我补一个测试,bug 会不会暴露; 如果我把任务切小,输出会不会更稳。没有因果框架,这些问题根本没法回答。

所以 coding agent 一旦不稳定,大家第一反应就是去找那根断掉的链条。

## 你以为自己面对的是一条直线

![一张对比图:左边是我们以为存在的简单因果链,右边是现实里真正运行的高噪声链条。](/diagrams/wrong-causal-graph.svg)

很多人脑子里的模型其实很简单:

`真实意图 -> SPEC -> Claude Code -> 正确代码`

这张图的问题不在于它完全错了,而在于它省略了太多中间变量。

它默认 SPEC 可以无损表达你的真实需求。它默认模型收到 SPEC 之后会像执行器一样严格落地,而不是用自己见过的常见模式去补空白。它还默认"代码看起来像对的"这件事,和"代码真的做对了"差不多是一回事。

这几个默认,现实里都站不住。

## 真正运行的那张图更长,也更脏

首先丢失的是意图。需求最完整的时候,其实只存在于你的脑子里。你写下来的 SPEC,已经是一次压缩。很多你觉得"不用写也该知道"的前提,都会在这一步蒸发掉。

接下来丢失的是上下文。仓库里原本就有一堆没写在 SPEC 里的约束:旧接口不能破,某个边缘行为得保留,某段代码虽然难看但暂时不能动,团队一直沿用某种错误处理方式。模型拿不到这些东西,就只能自己补。

而模型补空白时,用的不是你的真实系统,而是它训练里最常见的模式。所以最常见的结果不是胡说八道,而是另一种更危险的东西:局部都讲得通,拼起来却不是你要的那套行为。变量名很漂亮,注释也很像那么回事,真正跑起来还是偏。

最后一道漏口是验证。如果你没有先把需求变成可执行检查,没有跑测试,没有 build,没有让另一轮 review 去看"需求到底有没有被覆盖",那第一个真正严肃的 verifier 就只剩你自己,或者线上环境。

这样看,bug 不是一次意外。它更像是一套高噪声、多环节系统在约束不足时的正常产物。

## 真正能把成功率抬上去的,不是更会写 prompt

![一张闭环工作流图:先冻结 contract,再澄清、切分、写测试、实现、验证,任何一步失败都带着证据回退。](/diagrams/coding-agent-feedback-loop.svg)

一旦你把因果图重画出来,workflow 也得跟着换。

目标不再是"写出一条完美 prompt",而是"让系统很难在跑偏以后悄悄混过去"。

1. 先把 contract 冻住。目标是什么,非目标是什么,边界条件是什么,done 长什么样,都先写明白。
2. 不要让模型直接开写。先让它复述需求,把假设、风险点、受影响文件摊出来。
3. 任务切小。一个小场景一个小场景做,比一次吞完整个 feature 更容易验证。
4. 尽量把需求变成可执行检查。对逻辑改动来说,先写一个 failing test,往往比多加三段解释更有用。
5. 限制改动面。明确哪些文件能改,哪些不能动,顺手重构一律不算加分。
6. 每个 slice 都要跑验证。测试、build、lint、关键路径手验,缺哪个都不要说完成。
7. 写代码和判代码分开。实现者负责产出候选解,评审者负责抓 requirement coverage、回归风险和测试缺口。

这些动作不会把模型变成确定性的机器。它们做的,是把那些原本会悄悄溜到最后的"自由发挥",尽量提前暴露出来。

## 最该改的,其实是心里的角色定义

Prompt 工程当然有用。它能把平均表现往上抬一点。但平均值上去,不等于过程变稳。你真正想要的不是"这次写得更漂亮",而是"下一次不要突然跑偏到另一个宇宙去"。

稳定性来自控制。不是控制每一个 token,而是控制 contract、控制改动范围、控制验证节点、控制反馈回路的长度。

你手里并没有一台"SPEC 进去,正确代码出来"的机器。你手里是一个高方差的候选解生成器,被放进了一套工作流里。最后决定代码能不能上线的,不是模型,是这套闭环。

**模型负责产出代码。闭环决定这份代码算不算数。**
