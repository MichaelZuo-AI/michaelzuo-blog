把 [Andrej Karpathy 的 skills 仓库](https://github.com/forrestchang/andrej-karpathy-skills/blob/main/CLAUDE.md) 里的这四条规则粘贴到你的指令文件里。每一条都对应着模型默认行为中的一种具体失败模式。

## 1. 把你不知道的事情说出来

**失败模式:** 模型拿到一个有歧义的指令,默默选了一种解读,自信地执行下去。等你发现它选错了的时候,后续所有工作都得从错误的起点重来。

**规则:**
> 明确陈述你的假设。如果存在多种可能的解读,全部列出来——不要默默挑一个。如果有任何不清楚的地方,停下来问。

可靠不代表每次都对。可靠意味着你的不确定性在变成 bug 之前就被看见。

## 2. 不要增加没被要求的东西

**失败模式:** 你要一个函数,它给你一个类。你要一个类,它给你一个插件架构。每一行没被要求的代码,都是未来某一天会以没人预料到的方式出错的代码。

**规则:**
> 没要求的功能一律不加。单次使用的代码不做抽象。没被要求的"灵活性"或"可配置性"都不加。如果你写了 200 行,但其实 50 行就够,那就重写。

模型不是偷懒——它是在朝错误的方向发力。默认的 LLM 输出会奖励"看起来复杂",而可靠的 LLM 输出奖励的是克制。你必须在指令文件里把这点说清楚,否则模型会默认给出那个更适合做演示的版本。

## 3. 守住请求的边界

**失败模式:** 你要求改一行代码,结果 diff 动了十二个文件,因为模型顺手"改进"了周围的代码。

**规则:**
> 不要"改进"周围的代码、注释或格式。与现有风格保持一致,哪怕你觉得可以写得更好。每一行被改动的代码都必须直接对应到用户的请求。

验证这条规则起作用的方法:*你能在打开 diff 之前就预判它的形状吗?* 能,就说明模型是个可靠的协作者。不能,就说明它在做你没要求的事。

## 4. 开始之前就定义好"完成"

**失败模式:** 模型说一句 *"现在应该能跑了"*——没有测试、没有复现、没有证据。你运行代码,任务其实并没有完成。

**规则:**
> "修这个 bug" → "先写一个能复现 bug 的测试,再让它通过。"
> "加个校验" → "先写针对非法输入的测试,再让它通过。"

模糊的"完成"定义("让它能跑")会退化成一个澄清循环或一次瞎猜。具体的"完成"定义能让模型独立迭代,交付的是证据,而不是信心。

## 说明书上没写的那一部分

把上面这四条规则粘贴到你的 `CLAUDE.md` 里,下一次会话就会变得更好用。这是实用层面的回报。

但你把它们连起来读——*把你不知道的事情说出来,不要增加没被要求的东西,守住请求的边界,开始之前就定义好"完成"*——它们根本就不是 prompt engineering。这是一个资深工程师花好几年才能塞进初级工程师脑子里的四件事。初级工程师一旦做到这四点,大家就会从夸他"聪明"改成夸他"可靠"。

但就算 prompt 写得再完美,模型本质上还是概率性的。它依然会猜错、会跑偏、会凭感觉宣布胜利。打磨 prompt 只能把下限抬高,抬不动天花板。真正的工作在更上一层:**harness engineering**——在模型之外建立一套能用 ground truth 验证模型输出的反馈闭环。比如一个真的会跑起来的测试。Prompt 告诉模型要瞄准什么,harness 告诉你它有没有打中。

**规则带你走完大半程,反馈闭环带你走完剩下的。**
