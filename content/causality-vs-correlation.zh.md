很多“成功经验”之所以无法复制，不是因为你执行得不够好，而是因为：

**你学到的，本质只是相关性，而不是因果关系。**

这听起来像一句老生常谈。

但真正的问题是，我们在现实决策中几乎每天都在犯这个错误。

而最近一次让我真正意识到这一点的，是我在做 AI coding 的过程。

## 相关性 vs 因果性

我们每天看到的“经验总结”，大多是这样的结构：

- 做了 A
- 结果出现了 B

于是得出：

**A 导致了 B。**

但这其实只说明一件事：

**A 和 B 一起发生了。**

也就是相关性。

真正的因果关系要更严格一点：

**如果我主动改变 A，B 是否会稳定发生？**

或者换句话说：

**它是否可以被复现？**

![相关性和因果性的差别：相关性只是同时出现，因果性需要主动干预后仍然稳定。](/diagrams/correlation-causality-handdrawn.svg)

## 一个典型误判：AI coding

我最近在用 AI coding agent 做 agentic coding，直接生成 feature 级代码。

一开始的体验非常好：

- 开发速度显著提升
- 很多功能可以一次生成成功
- 一些重复性工作几乎被消灭

在这种体验下，很容易得出一个结论：

**AI coding 可以显著提高开发效率。**

这个结论并不荒谬。

它甚至有很多“证据”。

但问题在于，这些证据只证明了一件事：

我用了 AI coding，然后出现了几个成功结果。

这仍然只是相关性。

## 真正的问题出现在不可复现

继续使用一段时间之后，我开始遇到一些更真实的情况：

- 同样复杂度的需求，输出质量差异很大
- 有些代码在简单场景下可用，但在真实环境中失效
- 有时候需要花比手写更多的时间去修复问题

最关键的是：

**我无法稳定地复现“高质量输出”。**

这才是问题的核心。

不是 AI coding 从来没用。

而是它有时很好，有时很差。你很难提前知道这次会落在哪一边。

如果一个成功无法稳定复现，它就还不是方法。

它只是一次结果。

## 从能力问题到系统问题

这时候，很容易得出另一个错误结论：

**AI 还不够成熟，做不了复杂 coding。**

但这其实又是一个因果误判。

真正的问题不是简单地问：

**AI 能不能写代码？**

更重要的问题是：

**在什么条件下，它能稳定地写出符合预期的代码？**

模型能力当然重要。

但能力不是最终答案。

工程里真正关心的是：这份能力能不能被放进一个系统里，持续产出可验证、可交付、可维护的结果。

![AI coding 的错误短链和真实系统链条：模型能力之外，还有 spec、上下文、测试、选择和反馈。](/diagrams/ai-coding-system-handdrawn.svg)

## 相关性思维 vs 因果性思维

如果用相关性思维来看这个问题，你会看到：

- 我用了 AI
- 有时候成功
- 所以 AI 有用

这个判断没有完全错。

但它太粗。

如果用因果思维，你会继续问：

**在不同需求、不同上下文、不同代码库里，这个结果是否稳定？**

继续往下问，就会发现问题不只是模型能力。

真正的问题是：如何把模型能力约束成稳定产出。

## 从工具到系统：harness engineering

AI coding 本质上不是一个“用不用工具”的问题。

它是一个系统工程问题。

如果你只是写一个 prompt，然后直接生成代码，那你得到的本质上是一个带随机性的采样结果。

有时候采样很好。

有时候采样很差。

你不能把一次好的采样当成稳定能力。

要让它变成可复现的结果，你需要一套 harness engineering：

- 明确的 requirement 和 spec 约束
- 多轮生成与选择
- 自动化测试
- lint 和 static analysis
- 失败重试与反馈机制
- 对改动范围和验收条件的控制

这里的 harness engineering，不是写更长的 prompt。

它是把模型放进一个工程闭环里。

模型负责提出候选解。

harness 负责筛选、验证、反馈，并决定哪些候选解可以留下。

![Harness engineering 闭环：从 spec 到采样、选择、测试、静态检查、反馈重试，最后得到可交付代码。](/diagrams/harness-engineering-loop-handdrawn.svg)

## 本质是在做一件事

这套体系的目标不是让模型变成确定性机器。

模型本来就是概率系统。

真正要做的是：

**把随机成功，变成高概率成功。**

更准确一点说：

**裸用模型是在抽样；harness engineering 是在控制输出分布。**

这就是“用工程手段，把相关性压缩成接近因果的行为”。

你不是在证明模型永远可靠。

你是在构造一套条件，让它在这些条件下更稳定地产生你要的结果。

## 一个更准确的结论

一开始我以为：

**AI coding 可以提升开发效率。**

现在我会换一种说法：

**AI coding 加上合适的工程体系，才可能稳定提升开发效率。**

差别就在这里。

前者是一次经验总结。

后者才更接近一个可复现的系统设计。

## 这件事的普适意义

这个例子并不只属于 AI。

它其实是一个更通用的模式：

**你看到的很多“成功”，只是系统中某个局部的相关性。**

只有当你补齐整个系统，控制关键变量，设计验证机制之后，它才有机会变成可以复现的因果关系。

投资是这样。

产品增长是这样。

个人成长也是这样。

你复制不了的成功，往往不是因为你复制得不够像。

而是因为你复制的是表面行为，不是背后的因果系统。

## 最后一句话

**不能复现的成功，本质都是错觉。**

真正的能力，是把偶然的成功，变成可以被设计和重复的结果。
