软件不会自己进化。你上线了，它就停在你离开时的样子。每一次改进都需要人去发现差距、找到方案、动手实施。在你的两次干预之间，项目是冻结的。

如果不是这样呢？

我一直在思考「项目自我进化」意味着什么——持续地从生态系统中吸收模式，在没有人类主动推动的情况下变得更好。不是自愈（对故障做出反应），不是自测（验证正确性），而是自我进化：主动从社区学习，整合有效的东西。

`/always-hungry` 是我对这个想法的实现。它是一个 Claude Code 技能，给你的项目赋予自我进化模式。指向任何代码库，它会搜索开源社区中相关的模式，在你的代码上测试，然后合并通过验证的改动。自主地。反复地。每次运行，项目都会变好一点。

写这篇文章前 30 分钟，我刚在这个博客上跑了一次。它找到并应用了 2 个我没想到的改进。

## 自我进化循环

这个技能分三个阶段运行。三者组合形成一个循环：项目观察它的生态系统，用发现的东西做实验，然后吸收有效的部分。

![The Always Hungry Pipeline](/diagrams/always-hungry-pipeline.svg)

### 阶段一：侦察——观察生态系统

Agent 首先读你项目的 `CLAUDE.md`——就是你用来告诉 Claude Code 如何在项目中工作的那个文件。从中构建一份**项目画像**：项目是什么，处于什么领域，什么样的搜索查询能找到包含相关模式的仓库。

它搜索 GitHub，去重，过滤掉 fork 和长期未更新的仓库，然后跟 `seen.json` 状态文件比对，不会重复评估已处理过的仓库。对于通过筛选的仓库，它读 README、检查结构，然后问：*这里面有没有我的项目应该学习的东西？*

输出是一组**候选改进**——每个都包含来源仓库、目标文件和核心思路。

### 阶段二：评估——安全地实验

每个候选改进都会创建自己的 git 分支。Agent 读取现有代码，精准地应用改动，然后运行构建或测试命令。

构建失败？分支直接删除。不留痕迹。不破坏状态。实验失败——继续下一个。

构建通过？改动提交到评估分支。

这是自然选择的关卡。项目自己的质量标准决定什么能存活。Agent 不靠自己的判断来评估代码质量——它信任你的测试套件。你的测试就是免疫系统。

### 阶段三：应用——吸收有效的部分

通过评估的候选改进用 `--no-ff` 合并到 `main`（这样你能清楚看到项目学到了什么）。评估分支被清理掉。状态更新。一切推送到远程。

你回到项目，发现它已经进化了——干净的提交信息指向它学习的源仓库。

## 一次真实运行

我在你正在看的这个博客上跑了 `/always-hungry`。以下是自我进化循环产生的结果。

![Real Run: Scouting michaelzuo.vip](/diagrams/always-hungry-funnel.svg)

Agent 搜索了 Next.js 博客相关的 GitHub 仓库，扫描了 12 个仓库，发现其中 5 个是之前没见过的，然后根据「双语静态博客 + Next.js 15 + Tailwind CSS 4」的特征进行分级。

最终产生三个候选：

**1. JSON-LD Article 结构化数据**（来自 [leerob/next-mdx-blog](https://github.com/leerob/next-mdx-blog)）

博客之前没有结构化数据。搜索引擎能读 HTML，但无法以机器友好的方式理解文章元信息。Agent 在每篇文章页面加入了 `<script type="application/ld+json">`，包含 `BlogPosting` schema——标题、日期、作者、描述——全部使用代码库中已有的数据。零新代码路径，只是把现有数据接入了标准格式。

**2. 原生 Next.js sitemap.ts**（来自 [leerob/next-mdx-blog](https://github.com/leerob/next-mdx-blog)）

博客之前通过 Node.js postbuild 脚本生成站点地图——一个独立文件，在类型系统之外，构建后运行。Agent 用 Next.js 内置的 `app/sitemap.ts` 替换了它，生成同样的 XML，但类型安全、框架集成、不需要额外脚本。旧脚本被删除。

**3. robots.ts**（来自 [nelsonlaidev/nelsonlai.dev](https://github.com/nelsonlaidev/nelsonlai.dev)）

放弃。Agent 检查后发现 `public/robots.txt` 已存在且内容正确。无需操作。

两个应用的候选改进都独立通过了构建验证。博客在几分钟内完成了部署。

## 为什么是自我进化，而不是自我改进

市面上有很多「AI 代码改进」工具。大多数分析你的代码然后建议重构。那是自我改进——向内看。

自我进化不一样。它向**外部**看。它不问「你的代码有什么问题？」而是问「生态系统在做什么你还没吸收的？」改进来自真实的、经过生产验证的仓库——不是 LLM 在真空中推理出的最佳实践。

这个区别很重要，因为最好的改进不是你能从第一性原理推导出来的。它们是别人已经想到并经过实战检验的。JSON-LD 结构化数据不是你能通过「重构」代码发现的。它是你从看到别的博客这么做才学到的模式。

而且自我进化模式在设计上是保守的：

- **不引入新依赖。** 项目吸收的是模式，不是包。
- **构建门控。** 每个变异都必须通过项目自己的质量标准。
- **有状态。** 项目记住它看过什么。每次运行只发现真正的新内容。
- **精准。** 改动瞄准特定文件。不重写。没有范围蔓延。

## 身份系统

一个项目要进化，首先需要知道自己是什么。

第一次运行 always-hungry 时，它会生成一份**项目画像**——一个 JSON 文件，从生态系统的视角捕捉项目的身份。包括：

- 定制的搜索查询（如 `nextjs blog markdown static stars:>=500`）
- 按优先级排序的相关关键词
- 分级问题（如「这个仓库是否包含能改善博客 SEO、内容渲染或双语支持的模式？」）
- 测试命令和源码路径

这份身份跨运行持久化。项目在识别什么与自己相关这件事上越来越擅长。搜索空间收窄到它的实际领域。随着时间推移，候选改进更有针对性，命中率也在提高。

## 如何使用

安装技能后，在项目中打开 Claude Code，输入：

```
/always-hungry
```

就这样。项目读自己的 `CLAUDE.md`，按需构建身份，观察生态系统，做实验，吸收有效的东西。你也可以单独运行各阶段：

```
/always-hungry --scout-only    # 只观察，不实验
/always-hungry --dry-run       # 展示身份但不行动
/always-hungry --profile       # 重建项目身份
/always-hungry --show-last-run # 查看项目上次学到了什么
```

## 更大的想法

我们习惯把软件看作我们构建和维护的东西。Always-hungry 翻转了这个关系。项目维护它自己——或者更准确地说，它通过持续从生态系统学习来进化。

它做的是人类会无限期推迟的那种工作。添加结构化数据。把脚本替换成框架原生方案。采用六个月前在技术大会上看到但一直没时间实施的模式。这些事没一个是紧急的。但每一个都重要。

Always-hungry 持续做这些工作。改进是累积的。你的项目在你不主动工作的时候也在变好。

这才是真正的魔法。不是 agent 能写代码。而是**你的项目在你做别的事情的时候自己进化了**。
