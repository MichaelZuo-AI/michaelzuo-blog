我有个问题：项目永远没有「完成」的时候。总有一篇博客介绍了更好的模式，一个仓库用了更干净的方案，一个我还没采用的技术。开源社区迭代太快，靠人力根本跟不上。

所以我造了个 agent 来替我做这件事。

`/always-hungry` 是一个 Claude Code 技能，它能自主搜索开源社区、发现与你项目相关的改进、在你的代码上测试、然后合并通过验证的改动。全程零人工干预。不引入新依赖。每个改动在落地前都经过构建验证。

写这篇文章前 30 分钟，我刚在这个博客上跑了一次。它找到并应用了 2 个我没想到的改进。

## 工作原理

这个技能分三个阶段运行。每个阶段有清晰的输入输出和停止条件。

![The Always Hungry Pipeline](/diagrams/always-hungry-pipeline.svg)

### 阶段一：侦察（Scout）

Agent 首先读你项目的 `CLAUDE.md`——就是你用来告诉 Claude Code 如何在项目中工作的那个文件。从中提取技术栈、架构和领域信息，然后生成一份**项目画像**，包含针对性的 GitHub 搜索查询。

它搜索 GitHub，去重，过滤掉 fork 和长期未更新的仓库，然后跟 `seen.json` 状态文件比对（这样不会重复评估已处理过的仓库）。对于通过筛选的仓库，它会读 README、检查目录结构，并根据项目画像中的关联性问题进行分级。

输出是一组**候选改进**——每个都包含来源仓库、目标文件和核心思路。

### 阶段二：评估（Evaluate）

每个候选改进都会创建自己的 git 分支。Agent 读取现有代码，精准地应用改动，然后运行项目的构建或测试命令。

构建失败？分支直接删除。不留痕迹。不破坏状态。Agent 继续处理下一个。

构建通过？改动提交到评估分支。

这是关键的质量关卡。Agent 不靠自己的判断来评估代码质量——它信任你的测试套件。你的测试能抓住的问题，它就能拦住。测试抓不住的，它也拦不住。但至少，每个改动都是经过验证的。

### 阶段三：应用（Apply）

通过评估的候选改进用 `--no-ff` 合并到 `main`（这样你能清楚看到哪些提交来自 always-hungry）。评估分支被清理掉。`seen.json` 状态更新。一切推送到远程。

你回到项目，发现改进已经落地了，干净的提交信息指向源仓库。

## 一次真实运行

我在你正在看的这个博客上跑了 `/always-hungry`。以下是真实结果。

![Real Run: Scouting michaelzuo.vip](/diagrams/always-hungry-funnel.svg)

Agent 搜索了 Next.js 博客相关的 GitHub 仓库，扫描了 12 个仓库，发现其中 5 个是之前没见过的，然后根据「双语静态博客 + Next.js 15 + Tailwind CSS 4」的特征进行分级。

最终产生三个候选：

**1. JSON-LD Article 结构化数据**（来自 [leerob/next-mdx-blog](https://github.com/leerob/next-mdx-blog)）

博客之前没有结构化数据。搜索引擎能读 HTML，但无法以机器友好的方式理解文章元信息。Agent 在每篇文章页面加入了 `<script type="application/ld+json">`，包含 `BlogPosting` schema——标题、日期、作者、描述——全部使用代码库中已有的数据。零新代码路径，只是把现有数据接入了标准格式。

**2. 原生 Next.js sitemap.ts**（来自 [leerob/next-mdx-blog](https://github.com/leerob/next-mdx-blog)）

博客之前通过 Node.js postbuild 脚本生成站点地图——一个独立文件，在类型系统之外，构建后运行。Agent 用 Next.js 内置的 `app/sitemap.ts` 替换了它，生成同样的 XML，但类型安全、框架集成、不需要额外脚本。旧脚本被删除。

**3. robots.ts**（来自 [nelsonlaidev/nelsonlai.dev](https://github.com/nelsonlaidev/nelsonlai.dev)）

放弃。Agent 检查后发现 `public/robots.txt` 已存在且内容正确。无需操作。

两个应用的候选改进都独立通过了构建验证。博客在几分钟内完成了部署。

## 与众不同之处

市面上有很多「AI 代码改进」工具。大多数分析你的代码然后建议重构。Always-hungry 做的是根本不同的事：它向**外部**看。

它不问「你的代码有什么问题？」而是问「社区在做什么你还没采用的？」改进来自真实的、经过生产验证的仓库——不是 LLM 训练数据中关于最佳实践的泛泛建议。

而且它在设计上是保守的：

- **不引入新依赖。** Agent 用其他仓库的模式修改现有代码，但绝不添加包。
- **构建门控。** 每个改动必须通过你的构建/测试套件才能落地。
- **有状态。** 它记住看过什么，重复运行只发现真正的新内容。
- **精准。** 改动瞄准特定文件。不重写。没有「顺便也改一下...」的范围蔓延。

## 画像系统

第一次在项目上运行 always-hungry 时，它会生成一份画像——一个 JSON 文件，从 GitHub 搜索的视角捕捉你项目的身份。包括：

- 定制的搜索查询（如 `nextjs blog markdown static stars:>=500`）
- 按优先级排序的相关关键词
- 分级问题（如「这个仓库是否包含能改善博客 SEO、内容渲染或双语支持的模式？」）
- 你的测试命令和源码路径

画像跨运行持久化。Agent 的搜索效果会越来越好，因为搜索空间被限定在你项目的实际领域内。

## 如何使用

安装技能后，在项目中打开 Claude Code，输入：

```
/always-hungry
```

就这样。Agent 读 `CLAUDE.md`，按需生成画像，侦察、评估、应用。你也可以单独运行各阶段：

```
/always-hungry --scout-only    # 只搜索和报告
/always-hungry --dry-run       # 展示画像但不执行
/always-hungry --profile       # 重新生成项目画像
/always-hungry --show-last-run # 查看上次运行摘要
```

## 为什么要做这个

我相信 AI agent 最被低估的能力是**自主维护**。不是构建新功能——而是改进已有的东西。那种人类会无限期推迟的工作，因为它不够紧急所以排不上优先级，但又重要到不能忽视。

添加结构化数据。把脚本替换成框架原生方案。采用六个月前在技术大会上看到但一直没时间实施的模式。

Always-hungry 持续做这些工作。每次运行，它扫描社区中对你项目有益的模式，然后应用通过质量关卡的那些。改进是累积的。你的项目在你不主动工作的时候也在变好。

这就是魔法。不是 agent 能写代码。而是**你的项目在你做别的事情的时候自己变好了**。
