我最近在做一套端到端自动化的代码交付流程。最早那版很依赖 Claude Code 和 Opus。不是简单地把它当成默认模型而已。连任务怎么切、失败后怎么重试、什么情况下该继续往前跑，慢慢都按那套模型的行为习惯来设计了。

后来问题变得很明显。

从 Opus 4.5 到 4.7，这套流程里的表现开始往下掉。同样的任务，噪音更多，重试更多，改动稳定性更差，我介入得也更频繁。与此同时，GPT-5.4 在 Codex 里的表现，至少在我手头这类工作上，已经开始比我原来重度使用的 Claude Code 更好。

真正值得写下来的一点，不是谁暂时排第一。

真正的问题是，这次切换把我系统里的设计缺陷直接暴露出来了。如果领先模型一变，你的工作流就要跟着大改，那你做的不是自动化架构，而是一层包在某个模型外面的流程外壳。

## 排名还会继续变

很多团队现在还没把这件事当成基本前提。

模型榜单不会稳定。基准会换，版本会更新，真实场景里的适配度也经常比舆论更早变化。有的模型长程规划很强，过几个版本就开始飘。有的模型以前不擅长受控改代码，后来又追得很快。今天的第一名，不是一个可以写进系统设计文档里的常量。

所以目标不该是:先挑出冠军，再把整套流程围着它打磨。

更合理的目标是:默认冠军会换，而且切换成本要低。

如果每次排名变化都变成一次工程改造，那这套系统就没有给你带来杠杆，反而把你绑死了。

## 我第一版到底错在哪

不是某条 prompt 写坏了。是耦合藏得太深。

那版流程里，很多本来应该独立的东西，已经悄悄跟模型行为绑在一起了:

- 任务边界默认了某一种规划和执行节奏
- 重试逻辑默认了同一个模型多问几次就会变好
- 成功条件有不少还藏在 prompt 的措辞里
- 恢复路径依赖某个模型的工具调用习惯
- 编排层实际上是按一种失败模式调出来的

模型一直领先的时候，这些问题不明显。排名一动，隐性的耦合立刻全部变成维护成本。

prompt 不是接口。

模型也不是架构。

## 什么才叫 model agnostic

model agnostic 不是说所有模型都一样，更不是说你可以拿同一份 prompt 到处贴。

模型当然不一样。有的更会规划，有的更会收敛修改，有的工具纪律更强，有的就必须把任务切得更细。这些差异都是真的。

所谓 model agnostic，意思是把这些差异关在一层很薄的边界里，不要让它们渗进整套系统。

应该稳定下来的，是这些东西:

- 任务契约
- 上下文打包方式
- 工具接口
- 产物格式
- 验证规则
- 失败后的回退和升级策略

应该允许按模型调整的，是这些东西:

- provider 特定的 prompt 组织方式
- 上下文预算
- 工具调用的细节差异
- 选路策略
- 少量跟模型相关的恢复逻辑

边界能不能守住，决定了你到底是在做架构，还是在追模型脾气。

## 先把契约定稳

我现在更在意的是编排层和执行层之间的契约。

每次把任务交给模型，我希望交出去的是一个稳定的工作单元:

- 目标到底是什么
- 能用哪些工具
- 必须产出什么文件或结果
- 用什么验证来判断通过还是失败
- 如果这一步安全地做不完，什么时候升级给人

这套契约一旦清楚，模型就只是后面一个可以替换的执行器。上面这层流程不该因为 Claude、GPT，或者下一个模型谁排第一，就被迫重做一遍。

这跟基础设施设计是一个道理。你不会希望业务逻辑直接知道某个云厂商每个参数怎么命名。你会把差异收在边缘的一层 adapter 里。

## 不要让模型自己给自己验收

很多所谓的 model agnostic，最后都死在这里。

如果成功与否还是模型自己说了算，那系统本质上还是绑在那个模型身上。验收必须在模型外面。

对代码交付流程来说，这通常就是这些东西的组合:

- 测试
- 类型检查
- schema 校验
- 文件级 diff 约束
- 必需产物是否存在
- 政策检查，确认哪些能改，哪些不能改

模型可以提交方案，但不该自己宣布完工。

一旦验证独立出来，切模型这件事就会轻很多。你不再是在问:"我相不相信这个模型?" 你问的是:"它能不能在同一套契约和同一个 verifier 下把这步做完?"

这就是健康得多的边界。

## 把模型特定逻辑压薄

我当然还会做按模型调参。这很正常。

Claude 和 GPT-5.4 不需要同一套 prompt，也不需要相同的上下文预算、步长和重试阈值。model agnostic 不是把一份大 prompt 复制到多个 provider。

但这些差异应该只待在 adapter 附近。

如果你从一个领先模型切到另一个时，连任务图、产物 schema、日志结构、人工接管流程，甚至 verifier 都得改，那就说明模型特定逻辑已经漏到了不该漏的地方。

正确的感觉应该是:换驱动，不是换整辆车。

## 我现在采用的一条规则

我现在做这套流程，只带着一个前提:领先模型还会再换。

不是也许会换。是一定还会换。

所以我想要的，不是永远押对第一名，而是下次切换时尽量无聊:

- 工作流契约不要动
- 验证必须独立
- adapter 尽量薄
- 选路看实测表现，不看忠诚度
- 回退和晋升都要便宜

今天，在我自己的这套流程里，GPT-5.4 在 Codex 里的表现已经比我之前重度依赖的 Claude Code 和 Opus 更好。以后还会不会再变? 我默认会。

架构也该这样假设。

别把系统绑在今天最强的模型上。你真正该优化的，是明天第一名出现时，你能不能很轻松地切过去。
