反馈闭环才是关键

Agentic coding 真正有效，不是因为 prompt 更漂亮，而是因为反馈闭环里有 ground truth。

过去几个月，我几乎每天都在用 coding agent 做真实工程任务。反复用下来，我越来越确定一件事：agent 本身不是关键，围绕它的闭环才是关键。

最近读到 Karpathy 的 Sequoia Ascent 2026 总结，里面有一句话让我很有共鸣：

传统软件自动化的是你能明确写出来的东西。LLM 和强化学习自动化的是你能验证的东西。

这句话正好解释了我在 agentic coding 里反复看到的现象。

所以写这篇，算是把这段时间的思考记下来。

关键单元不是 prompt。不是模型。甚至不是 agent。

关键单元是反馈闭环。

闭环里有 ground truth，agent 才能把工作越做越好。闭环里只有感觉，agent 只能把输出写得更像那么回事。

Agentic coding 不是 prompt 魔法

很多人刚开始用 coding agent，会先优化最显眼的东西：prompt 写得更细，instructions 整得更完整，context 塞得更多，plan 拆得更清楚。

这些都有用。但它们不是工作流真正成立的原因。

Agentic coding 能跑起来，是因为代码本身处在一个会反驳它的环境里。编译器不在乎 agent 多自信。测试不会奖励漂亮叙述。runtime 不接受意图。应用要么行为正确，要么不正确。

这很稀有。

很多 LLM 工作流里，模型生成一个答案，然后人判断它看起来对不对。写代码不一样。模型生成一个改动，系统可以直接给出 ground truth。

build 失败。类型检查失败。单元测试失败。浏览器测试发现交互坏了。截图里文字挤在一起。日志证明那条路径根本没跑。

所以 coding agent 即使犯错也有价值。它的错误不只是错误。它是下一轮修正的材料。

闭环比 prompt 更重要

一个弱闭环，会把强模型变成很快的猜测器。

你让 agent “把这个做好一点”，但不给测试、不给验收标准、不给可观察行为、不给检查结果。它仍然会写代码。甚至会写出看起来合理的代码。但它没有被正确性推动。

强闭环会改变工作的性质。

带 ground truth 验证的 agentic coding 反馈闭环

Agent 得到目标，修改代码，环境检查改动，失败变成具体信号。下一次尝试就不再只靠语气和自信，而是靠证据。

这就是生成和工程的区别。

Agent 不是因为承诺自己会小心，所以变得可靠。它是因为闭环让粗心付出代价，所以变得有用。

Ground truth 和判断不是一回事

人类 review 仍然重要。品味仍然重要。架构仍然重要。安全边界仍然重要。

但人类判断不等于 ground truth。

Ground truth 是任务里那些能直接反驳你的部分。

测试要么过，要么不过。生成的网站里要么有这个 route，要么没有。migration 要么保住数据，要么没有。API 要么返回约定的 shape，要么没有。浏览器里按钮要么在容器内正常显示，要么没有。

人类判断决定什么应该为真。Ground truth 证明它是否真的为真。

好的 agentic engineering 会把这两件事分开。

人负责定义目标：什么重要，什么不能坏，什么 tradeoff 可以接受。闭环负责验证事实：改了什么，失败在哪里，什么仍然成立。

这两件事混在一起，agent 就有了空子。它可以用一个合理解释满足 reviewer，而不是用正确结果满足系统。

这就是 agentic coding 变软的地方。总结听起来对。diff 看起来忙。计划很完整。但 ground truth 没有真正压上去。

稳固的闭环是分层的

一个检查不够。

严肃的 coding loop 会叠很多层验证，因为每一层抓的是不同的失败模式。

静态检查抓语法、类型、lint、依赖边界和明显的契约漂移。

单元测试抓局部行为。

集成测试抓模块之间是否还对得上。

浏览器或 UI 检查抓用户看到的东西是否真的可用。

日志和 trace 抓预期路径有没有跑。

Diff review 抓可疑设计、范围膨胀、安全问题和别扭抽象。

生产指标抓本地环境看不到的问题。

这不是流程崇拜。它是在覆盖现实。

Agent 可以钻任何单一检查的空子。它可以让测试通过，但把设计弄坏。它可以满足 lint，但保留坏掉的 workflow。它可以给你一个看起来很干净的 diff，但浏览器里还是坏的。

分层验证会让逃逸路径变窄。

每一轮闭环都应该回答三个问题：

  • 这个改动做到了我们要的事吗？
  • 它有没有破坏我们在乎的东西？
  • 我们有证据，还是只有故事？

第三个问题最关键。

人的新工作是构建 evaluator

Agentic coding 改变了人的角色，但没有取消人的角色。

人不再是唯一的代码生产者。人变成了环境设计者：代码在这个环境里被生成、拒绝、修复、接受。

这意味着要写更好的 spec。把模糊需求变成检查。风险高的时候，先加测试，再让 agent 改。给 agent 可以执行的命令。视觉正确性重要时，就要截图。读 diff，而不是相信最后的 summary。

最强的 agentic engineer 不只是更会写 prompt。

他们更会构建 evaluator。

他们知道问题的哪些部分需要精确验证，哪些部分需要人的品味。他们知道什么时候测试通过是真的有意义，什么时候只是虚假的安慰。他们知道怎么把看不见的失败变成看得见的信号。

所以这个上限很高。

弱闭环里的单个 agent，只是一个聪明的 autocomplete。强闭环里的单个 agent，是在真实系统上做搜索。多个 agent 加上强闭环，就开始像一个有机械记忆的工程组织。

每次失败都可以变成测试。每条 review comment 都可以变成 instruction。每次事故都可以变成 invariant。每个重复错误都可以进入环境。

这才是复利。

模型重要。Prompt 重要。Context 重要。

但对 agentic coding 来说，真正长期有效的是闭环。

反馈闭环才是关键。
